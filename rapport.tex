\section{\textbf{Description du jeu Othello}}

Le jeu se joue sur un plateau de type \emph{othellier}, constitué d'une grille de huit cases par huit. Chaque joueur dispose de trente-deux pions de sa couleur, \emph{noir} ou \emph{blanc}. La position de départ est toujours identique : les pions noirs occupent les cases e4 et d5, tandis que les pions blancs se trouvent sur les cases d4 et e5. Le joueur possédant les pions noirs commence toujours la partie, puis les tours s'alternent entre les deux joueurs.

\begin{figure}[h]
    \centering
    \includegraphics[width=0.5\textwidth]{plateau_depart.png}
    \caption{Plateau de jeu avec les positions de départ}
    \label{fig:plateau_depart}
\end{figure}

\subsection{Règles de placement d'un pion}

Pour poser un pion sur le plateau, certaines conditions doivent être respectées. La case ciblée doit être \emph{vide}, et le pion doit permettre d'\emph{encadrer} au moins un ou plusieurs pions adverses en formant un \emph{sandwich} entre le nouveau pion et un pion déjà présent de la même couleur. Ce sandwich peut être réalisé verticalement, horizontalement ou diagonalement, et il ne doit jamais y avoir de cases vides entre les deux pions encadrants. Il est impossible de poser un pion sans capturer au moins un pion adverse, c'est-à-dire qu'un sandwich "noir/noir" ou "blanc/blanc" est interdit. Il est également possible de réaliser des sandwiches dans plusieurs directions lors d'un seul coup.

\subsection{Capture et conversion des pions}

Lorsqu'un pion encadre un ou plusieurs pions adverses, ceux-ci changent de couleur pour devenir de la couleur du joueur actif. Seuls les pions directement inclus dans le sandwich sont convertis, sans effet en chaîne sur les autres pions du plateau.

\subsection{Changement de joueur}

Le changement de joueur s'effectue normalement à chaque tour. Toutefois, si l'adversaire ne peut jouer aucun coup valide, le joueur actif rejoue immédiatement. Dans le cas contraire, l'adversaire doit obligatoirement effectuer son tour.

\subsection{Fin de partie}

La partie prend fin dans l'un des cas suivants : lorsque toutes les cases du plateau sont remplies, lorsqu'un joueur n'a plus de pion de sa couleur sur le plateau, ou lorsqu'aucun des deux joueurs ne peut poser de pion légalement, ce que l'on appelle un \emph{blocage}. Le vainqueur est alors le joueur possédant le plus grand nombre de pions de sa couleur au moment de l'arrêt de la partie.

\subsection{Règle optionnelle : le don de pion}

Il est également possible d'ajouter une règle optionnelle afin de contourner le cas du \emph{wipeout}, c'est-à-dire lorsque un joueur se retrouve sans pion. Dans ce cas, l'adversaire doit lui donner un de ses pions, que le joueur pourra placer sur une case libre de son choix afin de continuer la partie. Cette règle n'est pas systématique mais peut être utilisée pour rendre le jeu plus dynamique et éviter une fin trop brutale.

\section{\textbf{Architecture du projet et choix techniques}}

Nous avons choisi de réaliser ce projet en \emph{langage C}. Ce langage est reconnu pour sa rapidité et sa gestion efficace de la mémoire, ce qui est important pour les calculs de l'intelligence artificielle. Le programme sera divisé en plusieurs fichiers pour séparer les règles du jeu, les algorithmes de décision et l'interface avec l'utilisateur.

\subsection{Organisation des données}

Le plateau de jeu sera représenté par un tableau à deux dimensions de taille $8 \times 8$. Chaque case contiendra un entier pour indiquer si elle est \emph{vide}, ou si elle contient un pion \emph{noir} ou \emph{blanc}. 
Nous utiliserons par exemple les valeurs suivantes :

\begin{itemize}
  \item 0 pour une case vide,
  \item 1 pour un pion noir,
  \item -1 pour un pion blanc.
\end{itemize}

Cette structure simple nous permettra de copier facilement le plateau pour simuler des coups à l'avance.

\subsection{Structures de données}

Afin de faciliter la manipulation du plateau et des coups possibles, nous utiliserons plusieurs structures de données. Une structure \texttt{Plateau} permettra de regrouper l'état du plateau ainsi que les informations nécessaires au déroulement de la partie.

Une autre structure \texttt{Coup} pourra être utilisée pour représenter un coup possible, contenant par exemple les coordonnées de la case jouée ainsi que la liste des pions retournés. Cela permettra à l'intelligence artificielle de simuler facilement différents scénarios de jeu sans modifier directement le plateau principal.

Ces structures rendent le programme plus lisible et facilitent la maintenance du code en regroupant les informations liées au jeu dans des objets logiques.

\subsection{Parties du programme}

Le code sera organisé autour de trois modules principaux :

\begin{itemize}
    \item \textbf{Le moteur de jeu :} Il contiendra les fonctions pour vérifier si un coup est possible, retourner les pions capturés et déterminer quand la partie est terminée.
    \item \textbf{L'intelligence artificielle :} Ce module servira à calculer le meilleur coup à jouer pour l'ordinateur en analysant plusieurs coups à l'avance.
    \item \textbf{L'interface :} Elle s'occupera d'afficher la grille dans la console et de demander aux joueurs de choisir leurs cases.
\end{itemize}


\subsection{Déroulement d'une partie}

Le programme suivra une boucle principale représentant le déroulement de la partie. À chaque tour, les étapes suivantes seront exécutées :

\begin{enumerate}
    \item Affichage du plateau actuel.
    \item Calcul de la liste des coups valides pour le joueur actif.
    \item Si aucun coup n'est possible, le tour passe à l'adversaire.
    \item Sinon, le joueur (humain ou IA) choisit un coup.
    \item Le coup est appliqué et les pions capturés sont retournés.
    \item Le programme vérifie si la partie est terminée.
\end{enumerate}

Cette organisation permet de séparer clairement la logique de jeu, l'interaction utilisateur et les décisions de l'intelligence artificielle.

\section{\textbf{Algorithmes utilisés}}

L'objectif est de créer une intelligence artificielle capable de jouer contre un humain. Pour cela, nous allons utiliser les algorithmes suivants :

\subsection{Minimax et élagage Alpha-Beta}

Pour le mode "Joueur contre IA", nous utiliserons l'algorithme \emph{Minimax}. Son rôle est de simuler les tours suivants pour choisir le coup le plus avantageux pour l'ordinateur. Pour gagner du temps dans les calculs, nous ajouterons l'\emph{élagage Alpha-Beta}. Cela permet d'ignorer les mauvaises stratégies sans avoir à les explorer entièrement, ce qui rend l'IA plus rapide.

\subsection{Fonction d'évaluation}

Comme l'ordinateur ne peut pas prévoir tous les coups jusqu'à la fin de la partie, il utilisera une \emph{fonction d'évaluation}. Cette fonction donnera une "note" au plateau à un moment donné selon des critères stratégiques, comme le fait de posséder les coins ou d'avoir beaucoup de choix de coups possibles (la mobilité).

\subsection{Algorithme A* pour le mode solo}

Pour le mode \emph{solo}, nous utiliserons l'algorithme \emph{A*}. Il servira à résoudre des situations précises en cherchant le chemin le plus court pour atteindre un objectif, par exemple pour gagner avec le plus petit nombre de coups possible.

\section{\textbf{Analyse de la complexité}}

La plupart des fonctions qui gèrent le jeu Othello, comme vérifier si un coup est valide, jouer un coup ou générer les coups possibles, ont une complexité temporelle constante en $O(1)$. Cela vient du fait que le plateau fait toujours $8 \times 8$, donc le nombre d'opérations reste limité et ne dépend pas d'une taille variable.

En revanche, les fonctions liées à l'intelligence artificielle sont beaucoup plus coûteuses. Les algorithmes comme \emph{Minimax} explorent un arbre de jeu en testant plusieurs coups possibles à chaque étape. Cette complexité dépend de deux paramètres : $b$ et $d$. Le paramètre $b$ correspond au nombre moyen de coups possibles à un moment donné (le facteur de branchement), et $d$ est la profondeur de recherche, c'est-à-dire le nombre de coups que l'IA anticipe. Dans notre cas, cette profondeur est fixée à $d = 5$.

\subsection*{Complexité de Minimax}

La complexité temporelle de \emph{Minimax} est :

\begin{itemize}
    \item \textbf{Pire cas :} $O(b^d)$, lorsque tous les nœuds de l'arbre doivent être explorés sans aucune optimisation.
    \item \textbf{Cas moyen :} $O(b^d)$ également, car en pratique sans optimisation, la croissance reste exponentielle et proche du pire cas.
    \item \textbf{Meilleur cas :} $O(b^d)$, car sans élagage, Minimax explore systématiquement toutes les branches possibles.
\end{itemize}

\subsection*{Complexité de l'algorithme Alpha-Bêta}

L'algorithme \emph{Alpha-Bêta} améliore Minimax en supprimant les branches inutiles grâce aux bornes $\alpha$ et $\beta$ :

\begin{itemize}
    \item \textbf{Pire cas :} $O(b^d)$, lorsque aucun élagage n'est possible (ordre des coups défavorable).
    \item \textbf{Cas moyen :} environ $O(b^{\frac{3d}{4}})$, grâce à un élagage partiel dans la plupart des situations.
    \item \textbf{Meilleur cas :} $O(b^{\frac{d}{2}})$, lorsque les meilleurs coups sont explorés en premier, permettant un élagage maximal.
\end{itemize}

Au final, ce sont surtout ces algorithmes d'IA qui déterminent les performances du programme, alors que les autres opérations sur le plateau restent très rapides.

\section{\textbf{Défis techniques}}

La première difficulté du projet concerne la gestion des coups valides sur le plateau. En effet, lors du placement d'un pion, le programme doit vérifier les huit directions possibles (horizontales, verticales et diagonales) afin de déterminer si un \emph{sandwich} est formé entre deux pions de même couleur, sans qu'une case vide n'apparaisse entre eux.

De même, un autre défi réside dans la complexité de l'algorithme \emph{Minimax} utilisé pour l'intelligence artificielle. Le nombre de configurations possibles dans le jeu d'Othello est très important, ce qui peut entraîner un grand nombre de calculs. Pour réduire le temps de recherche, l'algorithme sera donc optimisé à l'aide de l'\emph{élagage Alpha-Beta}, qui permet d'éviter l'exploration de branches inutiles.

\section{\textbf{Organisation du groupe}}

Nous allons travailler ensemble sur la structure de base, puis nous répartirons les tâches pour la partie algorithmique. Les premières étapes seront de coder les règles du jeu et l'affichage simple, avant de passer au développement des intelligences artificielles et aux tests pour régler la difficulté du jeu.

\section{\textbf{Algorithmique et Intelligence Artificielle}}

\subsection{Implémentation du mode Joueur contre Joueur}

La première étape du projet a consisté à mettre en place une version fonctionnelle du jeu en mode \emph{joueur contre joueur}, sans intelligence artificielle. Cette étape a permis de valider l'ensemble des règles du jeu ainsi que la structure générale du programme.

\subsubsection{Boucle principale de jeu}

Le déroulement de la partie repose sur une boucle principale infinie qui simule l'alternance des tours entre les joueurs. À chaque itération, le programme :

\begin{enumerate}
    \item Affiche l'état actuel du plateau,
    \item Demande au joueur courant de saisir un coup,
    \item Vérifie la validité de ce coup,
    \item Applique le coup et met à jour le plateau,
    \item Change le joueur actif,
    \item Vérifie si la partie est terminée.
\end{enumerate}

Si le coup saisi est invalide, le programme redemande simplement une nouvelle entrée sans changer de joueur.

\subsubsection{Saisie des coups}

Les coups sont saisis sous forme d'une coordonnée alphanumérique (par exemple \texttt{D3}). Une fonction dédiée convertit cette entrée en indices exploitables dans le tableau :

\begin{itemize}
    \item La lettre est transformée en indice de colonne,
    \item Le chiffre est transformé en indice de ligne.
\end{itemize}

Ces coordonnées sont stockées dans une structure \texttt{Coup}, facilitant leur manipulation dans le programme.

\subsubsection{Validation des coups}

La fonction \texttt{coupValide} vérifie si un coup respecte les règles du jeu. Elle s'assure que :

\begin{itemize}
    \item La case visée est bien à l'intérieur du plateau,
    \item La case est vide,
    \item Au moins une capture est possible dans l'une des huit directions.
\end{itemize}

Pour cela, le programme parcourt les huit directions autour du coup grâce à un tableau de directions. La fonction \texttt{captureDirection} vérifie si un \emph{sandwich} est réalisable dans une direction donnée.

\subsubsection{Gestion des captures}

Lorsqu'un coup est valide, il est appliqué via la fonction \texttt{jouerCoup}. Celle-ci :

\begin{itemize}
    \item Place le pion du joueur courant sur le plateau,
    \item Parcourt les huit directions,
    \item Retourne les pions adverses capturés grâce à la fonction \texttt{retournerDirection}.
\end{itemize}

Cette dernière modifie directement les cases du plateau en remplaçant les pions adverses par ceux du joueur courant.

\subsubsection{Mise à jour des scores}

Après chaque coup, le programme recalcule le nombre de pions noirs et blancs présents sur le plateau en parcourant l'ensemble de la grille. Ces valeurs sont stockées dans la structure \texttt{Plateau} et utilisées pour déterminer le vainqueur en fin de partie.

\subsubsection{Changement de joueur}

Le changement de joueur est réalisé simplement en inversant la valeur du joueur courant (1 devient -1 et inversement). Cette approche permet une gestion efficace et compacte des tours.

\subsubsection{Détection de fin de partie}

Après chaque tour, le programme vérifie s'il reste au moins un coup valide possible. Pour cela, il teste toutes les cases du plateau. Si aucun coup n'est possible, la partie se termine et le programme affiche :

\begin{itemize}
    \item Le plateau final,
    \item Le score des deux joueurs,
    \item Le vainqueur ou un match nul.
\end{itemize}

Cette étape garantit que la partie respecte les conditions d'arrêt définies par les règles d'Othello.

Le cœur de notre projet repose sur le développement d'une intelligence artificielle capable d'analyser le jeu et de prendre des décisions stratégiques. Nous avons implémenté l'algorithme \emph{Minimax} associé à une fonction d'évaluation multicritères.

\subsection{L'algorithme Minimax}

Pour permettre à l'ordinateur d'anticiper les coups de son adversaire, nous avons développé l'algorithme \emph{Minimax}. Ce procédé fonctionne par récursion : l'IA simule ses propres coups (recherche du maximum) et les réponses optimales de l'adversaire (recherche du minimum) sur une profondeur de recherche définie.

Explication du fonctionnement de l'algorithme étape par étape: 
\begin{enumerate}
\item Exploration: on part de l'état actuel et on simule tous les coups possibles de MAX.
\item Simulation adverse : Pour chaque coup de MAX, on simule tous les coups possibles de MIN.
\item Feuilles: Quand on arrive au bout (ex: depth=0), on utilise la Fonction d'Évaluation pour donner un score au plateau.
\item Remontée:
\begin{itemize}
\item Si c'est au tour de MIN, il choisit le score le plus bas parmi ses enfants.
\item Si c'est au tour de MAX, il choisit le score le plus haut parmi ses enfants.
\end{itemize}
\end{enumerate}
Afin de ne pas altérer la partie en cours lors des simulations, le programme utilise des copies temporaires de la structure \texttt{Plateau}. À chaque étage de l'arbre de décision, l'IA génère la liste des coups possibles, applique le mouvement sur une copie de la grille, puis descend au niveau suivant. Une fois la profondeur maximale atteinte, la fonction d'évaluation est appelée pour attribuer un score au plateau final imaginé.
\medskip
Malgré la capacité de l'algorithme \textsc{Minimax} à prédire les prochains mouvements, le nombre de combinaisons possibles à Othello augmente de façon exponentielle en fonction de la profondeur de recherche. Afin d'améliorer ce processus et de donner à l'IA la possibilité de réfléchir plus loin dans le même intervalle de temps, nous avons mis en œuvre l'élagage Alpha-Béta. \hyperref[alg:minimax]{(cf annexe 10.1)}

\subsection{L'élagage Alpha-Béta}
Cet algorithme introduit deux bornes: 
\begin{itemize}
    \item Alpha $\alpha$ : Le meilleur score que MAX est assuré d'avoir jusqu'ici (initialisé à $-\infty$\ ).
    \item Beta $\beta$ : Le meilleur score que MIN est assuré d'avoir jusqu'ici (initialisé à $+\infty$\ ).
\end{itemize}
Eplication de l'algorithme étape par étape: 
\begin{enumerate}
\item Transmission des bornes : À chaque appel récursif, on transmet les valeurs actuelles de $\alpha$ et $\beta$ aux nœuds enfants.
\item Évaluation et Mise à jour :\begin{itemize}\item Le joueur \textsc{Max} met à jour $\alpha$ s'il trouve un coup offrant un meilleur score.
\item Le joueur \textsc{Min} met à jour $\beta$ s'il trouve un coup qui réduit le score.
\end{itemize}
\item La Coupure : Si à un moment donné $\alpha \ge \beta$, cela signifie que le joueur précédent ne choisira jamais cette branche, car il dispose déjà d'une meilleure option ailleurs. On arrête alors immédiatement l'exploration de ce sous-arbre.
\end{enumerate}
Cette méthode permet de réduire considérablement le nombre de nœuds visités. \hyperref[alg:alphabeta]{(cf annexe 10.2)}


\subsection{La fonction d'évaluation stratégique}

\begin{itemize}
    \item \textbf{Générateur de coups :} Cette fonction parcourt systématiquement le plateau pour lister toutes les cases vides respectant la règle du "sandwich" dans les huit directions possibles.
    \item \textbf{Gestion des captures :} La fonction \texttt{jouer\_coup} assure la mise à jour du plateau en retournant tous les pions adverses emprisonnés lors d'un mouvement.
    \item \textbf{Validation par tests unitaires :} L'ensemble de ces fonctions a été validé par une suite de tests unitaires indépendants, garantissant la fiabilité du calcul des scores et du respect des règles avant l'intégration finale.
    \item \textbf{La mobilité :} L'algorithme calcule la différence entre le nombre de coups possibles pour l'IA et pour l'adversaire. L'objectif est de restreindre les options de l'humain pour le forcer, à terme, à jouer des coups défavorables.

    \item \textbf{Le positionnement (Matrice de poids) :} Nous utilisons une matrice constante qui attribue une valeur à chaque case. Les coins sont les cases les plus valorisées (+100), tandis que les cases adjacentes aux coins reçoivent un malus car elles risquent d'offrir l'accès aux coins à l'adversaire.

    \item \textbf{Le matériel (Comptage des pions) :} Le nombre de pions est pris en compte, mais son importance est pondérée selon la phase de jeu. Il est secondaire en début de partie et devient le critère principal lors des derniers coups pour assurer la victoire finale.

\section{\textbf{Conclusion}}

Ce projet nous a permis de concevoir un jeu d'Othello complet en langage C, respectant toutes les règles officielles, ainsi qu'une interface qui laisse la possibilité de jouer en mode joueur contre joueur ou joueur contre intelligence artificielle.

L'objectif principal était de créer une intelligence artificielle capable de prendre des décisions stratégiques. Pour cela, nous avons utilisé l'algorithme \emph{Minimax}, optimisé par l'élagage \emph{Alpha-Bêta}. Ces méthodes nous ont permis d'effectuer la recherche dans un arbre de décision et la gestion de la complexité exponentielle. La fonction d'évaluation a aussi été très importante dans le comportement de l'IA. Elle combine plusieurs critères comme la mobilité, la stabilité des positions et le contrôle des coins, ce qui permet d'estimer de manière plus réaliste la valeur de chaque situation de jeu.

Nous avons également étudié l'algorithme A*, mais nous n'en n'avons pas eu l'utilité. En effet, A* est conçu pour résoudre des problèmes de recherche de chemin, comme des trajets ou des labyrinthes, alors qu'Othello est un jeu de stratégie à deux joueurs où chaque action dépend directement de l'adversaire. Il n'était donc pas adapté à notre projet.

Pour aller plus loin, plusieurs améliorations sont possibles pour enrichir le projet. Par exemple, on pourrait ajouter la règle du don de pion afin d'éviter certaines fins de partie trop rapides et de les rendre plus intéressantes. Du côté de l'intelligence artificielle, on pourrait  améliorer son efficacité en optimisant certaines parties du code, par exemple en organisant les coups explorés ou en évitant de recalculer certaines situations déjà rencontrées. Cela permettrait de rendre l'IA un peu plus rapide et plus efficace sans changer complètement son fonctionnement.

Ce projet nous a permis de construire une base solide d'un jeu complet en C, et il reste encore plusieurs pistes simples d'amélioration pour le rendre plus fluide et plus intéressant à jouer.

\subsection{Algorithme MiniMax}
\label{alg:minimax}

\begin{algorithm}[H]
\SetAlgoLined % Trace les lignes verticales
\DontPrintSemicolon % Cache les points-virgules en fin de ligne

\SetKwFunction{FMinimax}{MiniMax}

\BlankLine
\textbf{Fonction} \FMinimax{depth: int, isMax: bool} : \textbf{entier}\;
\Debut{
    \Si{game\_over \textbf{ ou } depth == 0}{
        \Retour évaluation\_plateau()\;
    } 

    \Si{isMax}{
        bestScore $\leftarrow$ $-\infty$\;
        \Pour{chaque coup $m$ possible}{
            Faire move $m$\;
            score $\leftarrow$ \FMinimax{depth - 1, FAUX}\;
            Défaire move $m$\;
            \Si{score $>$ bestScore}{
                bestScore $\leftarrow$ score\ \tcc{Si depth == max, mémoriser m}
            }
        } 
    }
    \Sinon{
        bestScore $\leftarrow$ $+\infty$\;
        \Pour{chaque coup $m$ possible}{
            Faire move $m$\;
            score $\leftarrow$ \FMinimax{depth - 1, VRAI}\;
            Défaire move $m$\;
            \Si{score $<$ bestScore}{
                bestScore $\leftarrow$ score\;
            }
        }
    } 
    
    \Retour bestScore\;
}
\caption{Algorithme Minimax}
\end{algorithm}

\subsection{Algorithme Alpha-Béta}
\label{alg:alphabeta}

\begin{algorithm}[H]
\SetAlgoLined
\DontPrintSemicolon
\SetKwFunction{FAlphaBeta}{AlphaBeta}
\SetKw{Ou}{ou}
\SetKwBlock{Debut}{début}{fin}

\textbf{Fonction} \FAlphaBeta{depth: int, $\alpha$: int, $\beta$: int, isMax: bool} : \textbf{entier}\;
\Debut{
    \Si{game\_over \Ou depth $\le$ 0}{
        \Retour evaluation\_plateau()\;
    } \textbf{Fin si}\;

    \Si{isMax}{
        \Pour{chaque move $m$ possible}{
            Faire move $m$\;
            score $\leftarrow$ \FAlphaBeta{depth - 1, $\alpha$, $\beta$, FAUX}\;
            Defaire move $m$\;
            \Si{score $>$ $\alpha$}{
                $\alpha$ $\leftarrow$ score\;
                bestMove $\leftarrow$ $m$\;
                \Si{$\alpha$ $\ge$ $\beta$}{
                    \textbf{break} \tcc*[r]{Coupure Alpha}
                } 
            } 
        } 
        \Retour $\alpha$\;
    }
    \Sinon{
        \Pour{chaque move $m$ possible}{
            Faire move $m$\;
            score $\leftarrow$ \FAlphaBeta{depth - 1, $\alpha$, $\beta$, VRAI}\;
            Defaire move $m$\;
            \Si{score $<$ $\beta$}{
                $\beta$ $\leftarrow$ score\;
                bestMove $\leftarrow$ $m$\;
                \Si{$\alpha$ $\ge$ $\beta$}{
                    \textbf{break} \tcc*[r]{Coupure Béta}
                } 
            } 
        } 
        \Retour $\beta$\;
    }
}
\caption{Algorithme Alpha-Béta}
\end{algorithm}
